\# Артефакт 12: Описание сценариев использования приложения



\## **Сценарий 1. Пользователь записывает тренировку (основной поток)**

**Актёр**: Пользователь с часами Garmin / Apple Watch / телефоном

**Предусловие:** авторизован, устройство сопряжено (через HealthKit или BLE)

**Триггер:** завершение тренировки на устройстве



**Шаги:**

1. Устройство отправляет POST /api/v1/trainings с данными:{ "type": "run", "distance\_m": 5000, "duration\_sec": 1800, "avg\_hr": 145, "gpx\_track": "...", "idempotency\_key": "uuid-123" }
2. API Gateway проверяет JWT и rate limit
3. Модуль Training сохраняет тренировку в PostgreSQL (user\_id, type, distance, duration, track)
4. Модуль Training публикует событие training.completed в Kafka
5. Асинхронно (поток 1): Gamification Service проверяет рекорд → если да, публикует achievement.earned
6. Асинхронно (поток 2): Notification Service отправляет push друзьям (через Firebase / APNS)
7. Асинхронно (поток 3): Analytics Service обновляет агрегаты в ClickHouse
8. Пользователь получает ответ 201 Created + краткое сравнение с прошлой тренировкой



**Альтернативный поток (дубликат):**

Если idempotency\_key уже был → ответ 200 OK с ранее сохранённой тренировкой



**Альтернативный поток (ошибка валидации):**

Если дистанция отрицательная или пульс > 250 → 400 Bad Request







\## **Сценарий 2. Поиск людей для совместной тренировки**

**Актёр:** Пользователь

**Предусловие:** включена геолокация, дано согласие на обработку

**Триггер:** нажатие «Найти людей рядом»



Шаги:

1. Мобильное приложение отправляет текущие координаты (широта, долгота, точность)
2. API Gateway → модуль Social → Redis (Geohash)
3. Redis возвращает user\_id активных пользователей в радиусе 1 км, обновлявших позицию за последние 10 минут
4. Модуль Social обогащает список: аватарка, вид спорта, уровень, счётчик совместных тренировок
5. Пользователь выбирает «Создать событие» → бег в парке через 15 минут
6. Модуль Social публикует group.event.created в Kafka
7. Notification Service отправляет push всем найденным пользователям



**Альтернативный поток (нет людей рядом):**

Ответ 200 OK с пустым списком + предложение расширить радиус







\## **Сценарий 3. Получение персонализированной промоакции**

**Актёр**: Пользователь

**Предусловие**: есть завершённые тренировки за последние 7 дней

**Триггер**: завершение тренировки (бег) + наличие активной акции в регионе



**Шаги**:

1. Модуль Promotion Service подписан на training.completed из Kafka
2. Проверяет: тип тренировки = бег, дистанция > 5 км, регион пользователя = Москва
3. Находит активную акцию: «Скидка 20% на кроссовки Nike в Москве»
4. Отправляет push-уведомление с deep link на магазин
5. Логирует факт отправки для аналитики



**Альтернативный поток (нет акции)**: ничего не отправляется







\## **Сценарий 4. Сравнение с профессионалом**

**Актёр**: Пользователь

**Триггер**: открытие экрана «Сравнение»



**Шаги**:

1. Модуль Training запрашивает последние 7 тренировок пользователя из PostgreSQL
2. Запрашивает агрегаты по региону из ClickHouse (средний темп, дистанция, частота)
3. Запрашивает эталонные показатели профи (отмеченный аккаунт)
4. Формирует ответ:{ "vs\_self": "+5% к выносливости", "vs\_region": "вы быстрее 68% бегунов", "vs\_pro": "на 15% медленнее" }
