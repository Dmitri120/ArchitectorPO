\# Артефакт 11: Список ADR



\## **ADR-001: Выбор модульного монолита как стартовой архитектуры**



**Контекст**

Нужно запустить приложение за 3–4 месяца с малым бюджетом, но с возможностью масштабирования до миллионов пользователей. Команда — 5–7 разработчиков.



**Рассмотренные альтернативы:**

* Чистые микросервисы (K8s, 10+ сервисов)
* Классический монолит (один деплой)
* Serverless (AWS Lambda)



**Решение**

Модульный монолит на Python/FastAPI с жёсткой границей между модулями (тренировки, соцсеть, геймификация, инвентарь, промо). Внутри монолита - синхронные вызовы через Python-функции (быстро). Между модулями - асинхронные события через Kafka.



**Последствия:**

* Плюсы: быстрый старт, одна кодовая база, простое тестирование
* Минусы: нельзя деплоить модули независимо, риск «сцепления»
* Компромисс: модули выделены слабо, но позже их можно «вырезать» в микросервисы



**Статус:** принято







\## **ADR-002: Использование PostgreSQL как основной БД + Redis для геоданных**



**Контекст**

Нужно хранить: профили, тренировки, социальные группы, инвентарь. Также нужен быстрый поиск «люди рядом» (геопоиск).



**Рассмотренные альтернативы**

* MongoDB (гибкая схема, но слабые JOIN)
* ClickHouse как основная (отлично для аналитики, плохо для транзакций)
* Только PostgreSQL с PostGIS



**Решение**

* PostgreSQL — всё, что требует ACID и связей (тренировки, группы, пользователи, инвентарь)
* Redis + Geohash — кеш геопозиций активных пользователей (TTL 5–10 минут)
* ClickHouse — отдельно для аналитики (агрегаты по тренировкам, сравнения с регионом)



**Последствия**

* Плюсы: надёжность, скорость геопоиска, разделение OLTP и OLAP
* Минусы: два дополнительных компонента (Redis, ClickHouse)
* Компромисс: геоданные живут в Redis не более 10 минут — точность снижена, но нагрузка на БД минимальна



**Статус:** принято







\## **ADR-003: Событийная шина (Kafka) для асинхронных сценариев**



**Контекст**

Рекорды, награды, уведомления друзьям, обновления аналитики — не должны тормозить основной поток записи тренировки.



**Рассмотренные альтернативы**

* RabbitMQ (проще, но слабее для replay и высокой нагрузки)
* Прямые вызовы в коде (сделать всё синхронно)
* AWS SQS + SNS (привязка к вендору)



**Решение**

Apache Kafka — топологи: trainings, achievements, notifications, analytics.

Событие «тренировка сохранена» → триггерит:

* проверку рекорда (Gamification Service)
* отправку уведомлений друзьям (Notification Service)
* обновление агрегатов в ClickHouse



**Последствия**

* Плюсы: надёжность, replay, масштабирование
* Минусы: сложность настройки, Zookeeper/KRaft
* Компромисс: для MVP можно заменить Redis Pub/Sub, но сразу закладываем Kafka



**Статус**: принято







\## **ADR-004: API Gateway с маршрутизацией и rate limiting**



**Контекст**

Мобильные клиенты, веб, устройства (BLE, MQTT). Нужна единая точка входа, защита от брутфорса, ограничение запросов.



**Рассмотренные альтернативы**

* Встроенный Gateway в приложение (FastAPI сам)
* Nginx + Lua
* Kong / Traefik / Ambassador



**Решение**

Kong Gateway (Enterprise) или Traefik (open source).

Функции:

* JWT-аутентификация
* Rate limiting: 1000 запросов/час на пользователя, 100/час на геопоиск
* Маршрутизация на модули монолита
* Логирование всех запросов в Kafka



**Последствия**

* Плюсы: безопасность, контроль, централизованные логи
* Минусы: дополнительный компонент, задержка примерно 5–10 мс
* Компромисс: на старте можно обойтись Ingress + nginx, но закладываем Gateway



**Статус**: принято







\## **ADR-005: Идемпотентность при записи тренировок**



**Контекст**

Устройства могут дублировать данные (сбой сети, повторная отправка). Нельзя создать дубликат тренировки.



**Рассмотренные альтернативы**

* Уникальный ключ (user\_id + timestamp + тип тренировки) — ненадёжно
* Клиент генерирует UUID и проверяем на сервере



**Решение**

Клиент (телефон/часы) отправляет idempotency\_key (UUIDv4) в заголовке Idempotency-Key. Сервер хранит ключи в Redis (TTL 24 часа) и возвращает уже сохранённую тренировку при повторной отправке.



**Последствия**

* Плюсы: защита от дублей
* Минусы: дополнительный Redis lookup
* Компромисс: TTL 24 часа — если пользователь отправит ту же тренировку через 2 дня, создастся новая 



**Статус:** принято
